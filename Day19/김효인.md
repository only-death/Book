# 23. 실행 컨텍스트

: 자바스크립트가 스코프를 기반으로 식별자와 식별자에 바인딩된 값(식별자 바인딩)을 관리하는 방식과 호이스팅이 발생하는 이유, 클로저의 동작 방식, 그리고 태스크 큐와 함께 동작하는 이벤트 핸들러와 비동기 처리의 동장 방식을 이해할 수 있다.

1. 스코프를 기반으로 식별자와 식별자에 대한 바인딩된 값을 관리하는 방식
2. 호이스팅이 발생하는 이유
3. 클로저의 동작 방식
4. 테스트 큐와 함께 동작하는 이벤트 핸들러
5. 비동기 처리의 동작 방식

## 1. 소스코드의 타입

| 코드 유형                     | 설명                                                                                                                                                                                                                       |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **전역 코드 (Global Code)**   | 최상위 스코프에서 실행되는 코드로, 함수나 모듈 내부가 아닌 코드. 전역 컨텍스트에서 실행되며 `var`로 선언된 변수는 전역 객체(window, globalThis 등)의 속성이 됨. 전역에 정의된 함수, 클래스 등의 내부 코드는 포함되지 않음. |
| **함수 코드 (Function Code)** | 함수 내부에서 실행되는 코드로, 함수 호출 시 실행됨. 독립적인 실행 컨텍스트를 가지며, 지역 변수와 `this` 바인딩이 결정됨. 함수 내부에 중첩된 함수, 클래스 등의 내부 코드는 포함되지 않음.                                   |
| **eval 코드 (Eval Code)**     | `eval()` 함수를 통해 동적으로 실행되는 코드. 실행 시 별도의 실행 컨텍스트를 생성하며, 보안 및 성능 이슈로 인해 사용이 권장되지 않음.                                                                                       |
| **모듈 코드 (Module Code)**   | `import`, `export`를 사용할 수 있는 코드로, ES6+에서 도입된 모듈 시스템을 따름. 모듈은 기본적으로 **strict mode**에서 실행됨. 모듈 내부의 함수, 클래스 등의 내부 코드는 포함되지 않음.                                     |

## 2. 소스코드의 평가와 실행

: "소스코드의 평가"와 "소스코드의 실행"

- 소스코드 평가
  - 실행 컨텍스트를 생성하고
  - 변수, 함수 등의 선언문만 먼저 실행
  - 생성된 변수나 함수 식별자를 키로 실행 컨텍스트가 관리하는 스코트(렉시컬 환경 레코드)에 등록
- 소스 코드의 실행(런타임)
  - 소스코드 실행에 필요한 정보(변수나 함수의 참조를 실행 컨텍스트가 관리하는 스코프에 검색)를 취득한다
  - 변수 값의 변경 등 소스코드의 실행 결과는 다시 컨텍스트가 관리하는 스코프에 등록

```js
//소스코드 평가
// x는 실행 컨텍스트가 관리하는 스코프에 등록
// undefined로 초기화된다
var x = 1;

// 소스코드 실행
// var은 평가과정에서 실행이 완료됨
// 변수 할당문 x = 1; 만 실행하기 위해 실행 컨텍스트에 x 변수가 선언된 변수인지 확인
// 선언된 변수라면 값을 할당
// 실행 컨텍스트에 결과를 등록하여 관리
x = 2;

console.log(x); // 2
```

## 3. 실행 컨텍스트의 역할

1. 전역 코드 평가
   1. 선언문만 먼저 실행(변수 선언문, 함수 선언문)
   2. 전역 스코프 등록
   3. var 키워드로 선언된 전역 변수와 함수 선언문으로 정의된 전역 함수는 전역 객체의 프로퍼티와 메서드가 됨
2. 전역 코드 실행
   1. 전역 변수에 값이 할당되고 함수가 호출됨
   2. 함수가 호출되면 전역 코드의 실행을 멈추고 함수 내부로 진입
3. 함수 코드 평가
   1. 매개변수와 지역 변수 선언문 실행
   2. 실행 컨텍스트가 관리하는 지역 스코프에 등록
   3. 함수 내부에 지역 변수처럼 사용하는 arguments 객체 생성
   4. 지역 스코프에 arguments 등록
   5. this에 바인딩
4. 함수 코드 실행(런타임 실행)
   1. 매개변수와 지역 변수에 값이 할당
   2. console.log 메서드 호출
      1. console 식별자는 전역 객체에 프로퍼티로 존재하므로 전역 스코프를 통해 검색
      2. log 프로퍼티를 console 객체의 프로토타입 체인을 통해 검색
      3. console.log(a + b + c) 메서드에 인수를 전달
      4. 스코프 체인을 통해 a, b, c 검색
   3. console.log 메서드의 실행이 종료되면 전역 코드 실행을 계속한다

**실행 컨텍스트는 소스코드를 실행하는 데 필요한 환경을 제공하고 코드의 실행 결과를 실제로 관리하는 영역이다**

1. 선언에 의해 생성된 모든 식별자를 스코프를 구분하여 등록하고 상태 변화를 지속적으로 관리

   - 실행 컨텍스트는 **식별자(변수, 함수 등)를 스코프 단위로 관리**하여 코드 실행 시 필요한 정보를 제공한다.
   - 전역 코드 평가 단계에서 **변수와 함수 선언문을 먼저 실행**하여 전역 스코프에 등록한다.
   - `var` 키워드로 선언된 변수와 함수 선언문으로 정의된 함수는 **전역 객체의 프로퍼티와 메서드**가 된다.

2. 스코프는 중첩 관계에 의해 스코프 체인을 형성해야 함 즉, 스코프 체인을 통해 상위 스코프로 이동하며 식별자를 검색할 수 있어여 함

   - 변수나 함수 호출 시 **현재 스코프에서 찾고, 없으면 상위 스코프로 이동하며 검색**한다.
   - 즉, **스코프 체인을 통해 필요한 식별자를 찾을 수 있다.**
   - 예를 들어 `console.log(a + b + c)`를 실행할 때:

     1. `console`은 전역 객체의 프로퍼티이므로 **전역 스코프에서 검색**된다.
     2. `log`는 `console` 객체의 프로토타입 체인을 통해 검색된다.
     3. `a, b, c`는 **스코프 체인을 통해 순차적으로 검색**된다.

     ```js
     function outer() {
       let a = 10;
       function inner() {
         let b = 20;
         console.log(a + b); // 🔹 스코프 체인을 따라 a를 찾음
       }
       inner();
     }
     outer();
     ```

3. 현재 실행 중인 코드의 실행 순서를 변경할 수 있어야 하며 다시 되돌아갈 수 있어야 함

   - **함수 호출이 발생하면 실행 컨텍스트가 새로운 컨텍스트를 생성하고, 기존 실행 컨텍스트는 잠시 멈춘다.**
   - 함수 실행이 끝나면 실행 흐름이 원래 위치로 되돌아간다.
   - 즉, 실행 컨텍스트는 **콜 스택을 이용해 함수 실행 순서를 조정**한다.

   ```javascript
   function first() {
     console.log('First');
   }

   function second() {
     console.log('Second');
     first(); // 🔹 first() 실행 -> 현재 실행 중인 second()는 잠시 멈춤 console.log("Second End"); } second(); console.log("Global End");
   }
   ```

   - **실행 순서**
     1. `second()` 실행 → `"Second"` 출력
     2. `first()` 호출 → `"First"` 출력
     3. `first()` 종료 → `second()`로 복귀 → `"Second End"` 출력
     4. `second()` 종료 → 전역 코드 실행 → `"Global End"` 출력

## 4. 실행 컨텍스트 스택

**스택의 특징**

1. **후입선출 (LIFO, Last In First Out)**
   - 나중에 들어온 데이터가 먼저 나가는 구조
   - 마치 **접시를 쌓아 올리고, 위에서부터 하나씩 꺼내는 것**과 같음
2. **함수 호출 스택(Call Stack)과 실행 컨텍스트**
   - JavaScript의 실행 컨텍스트는 **콜 스택(Call Stack)**에서 관리
   - 함수가 호출되면 **새로운 실행 컨텍스트가 스택의 맨 위에 추가(push)되고**
     함수 실행이 끝나면 **스택에서 제거(pop)되며 원래 실행 위치로 돌아감**

```js
const x = 1;

function foo() {
  const y = 2;

  function bar() {
    const z = 3;
    console.log(x + y + z);
  }
  bar();
}

foo();
```

**1. 전역 코드 평가와 실행**
**① 전역 코드 평가**

- 전역 실행 컨텍스트 생성 후 **실행 컨텍스트 스택(Call Stack)에 push**
- **식별자 등록 (변수 및 함수 선언)** - `x` (전역 변수) → `undefined`로 초기화 - `foo` (함수 선언) → 메모리에 저장
  **② 전역 코드 실행**
- `x = 1` 값 할당
- `foo()` 호출 → **새로운 실행 컨텍스트 생성 후 스택에 push**

  **2. `foo` 함수 코드 평가와 실행**
  **① `foo` 함수 평가**

- `foo` 실행 컨텍스트 생성 후 **실행 컨텍스트 스택에 push**
- **식별자 등록 (변수 및 중첩 함수 선언)**
  - `y` (지역 변수) → `undefined`로 초기화
  - `bar` (중첩 함수) → 메모리에 저장
    **② `foo` 함수 실행**
- `y = 2` 값 할당
- `bar()` 호출 → **새로운 실행 컨텍스트 생성 후 스택에 push**

**3. `bar` 함수 코드 평가와 실행**
**① `bar` 함수 평가**

- `bar` 실행 컨텍스트 생성 후 **실행 컨텍스트 스택에 push**
- **식별자 등록 (변수 선언)**
  - `z` (지역 변수) → `undefined`로 초기화
    **② `bar` 함수 실행**
- `z = 3` 값 할당
- `console.log(x + y + z)` 실행 - **스코프 체인**을 통해 `x`, `y`, `z` 값을 찾음 - 결과 출력: `6`
  **③ `bar` 함수 실행 종료 → 실행 컨텍스트 스택에서 pop**
- 실행 컨텍스트 스택에서 **`bar` 실행 컨텍스트 제거**

**4. `foo` 함수 실행 종료**

- `foo`의 실행이 끝나면서 실행 컨텍스트 스택에서 **`foo` 실행 컨텍스트 제거 (pop)**

**5. 전역 코드 실행 종료**

- 모든 코드가 실행 완료되면서 **전역 실행 컨텍스트 제거 (pop)**
- 실행 컨텍스트 스택이 **완전히 비워짐**

### 5. 렉시컬 환경

: 변수를 저장하고, 접근할 수 있도록 관리하는 객체

- 환경 레코드
  - 현재 실행 중인 스코프에서 선언된 **변수와 함수**를 저장하는 곳
  - `let`, `const`, `var`로 선언한 변수, `function` 키워드로 정의한 함수 등이 저장
- 외부 렉시컬 환경에 대한 참조
  - 현재 환경이 상위(부모) 환경을 참조하는 부분
  - 즉, **스코프 체인(Scope Chain)**을 형성\*\*

### 6. 실행 컨텍스트의 생성과 식별자 검색 과정

1. 전역 객체 생성
   - 전역 객체도 Object.prototype을 상속받는다
2. 전역 코드 평가
   1. 전역 실행 컨텍스트 생성
      1. 실행 컨텍스트에 push
   2. 전역 렉시컬 환경 생성
      1. 환경 레코드, 외부 렉시컬 환경에 대한 참조 구성
         1. 전역 환경 레코드 생성
            - 전역 변수를 관리하는 전역 스코프, 전역 객체의 빌트인 전역 프로퍼티와 빌트인 전역 함수, 표준 빌트인 객체를 제공한다
            - let const 키워드로 선언한 전역 변수는 전역 프로퍼티가 되지 않고 개념적인 불록 내에 존재
            - var 전역변수와 let, const 전역 변수를 구분하기 위해 **전역 환경 레코드는 객체 환경 레코드(var, 함수, 표준 빌트인)와 선언적 환경 레코드(let, const)**로 구성되어 있다
            1. **객체 환경 레코드**(var, 함수, 표준 빌트인)
               - **BindingObject**를 통해 **객체의 프로퍼티와 메서드**가 됨
               - 전역 객체를 가리키는 식별자(window) 없이 전역 객체의 프로퍼티를 참조(window.alert의 alert 참조)할 수 있는 메커니즘
               - 변수 호이스팅과 함수 호이스팅의 차이
                 - 변수 호이스팅: 변수 선언문 이전에 참조한 변수의 값은 언제나 undefined
                 - 함수 호이스팅: BindingObject를 통해 전역 객체의 키로 등록, 생성된 함수 객체를 즉시 할당
            2. 선언적 환경 레코드 할당(let, const)
               - 일시적 사각지대
      2. this 바인딩
         - `[[GlobalThisValue]]` 내부 슬롯에 this 가 바인딩되고, this 바인딩은 **전역 환경 레코드**와 **함수 환경 레코드**에만 존재
      3. 외부 렉시컬 환경에 대한 참조 결정
         - 상위 스코프를 가리키고 이를 통해 단방향 링크드 리스트인 스코프 체인을 구현
   3. 전역 코드 실행
      - 식별자 결정: 어느 스코프의 식별자를 참조하는지 결정
   4. foo 함수 코드 평가
      1. 함수 실행 컨텍스트 생성
      2. 함수 렉시컬 환경 생성
         1. 함수 환경 레코드 생성
            - 매개변수, arguments 객체, 함수 내부에서 선언한 지역변수화 중첩 함수 등록, 관리
         2. this 바인딩
            - `[[ThisValue]]` 내부 슬롯에 바인딩
         3. 외부 렉시컬 환경에 대한 참조 결정
            - **어디서 호출했는지가 아닌 어디에 정의했는지에 따라 상위 스코프를 결정함**
   5. foo 함수 코드 실행
      - 식별자 결정을 위해 **실행 중인 컨텍스트의 렉시컬 환경에서 식별자를 검색하기 시작**
        - 실행 컨텍스트의 렉시컬 환경 -> 외부 렉시컬 환경
   6. bar 함수 코드 평가
   7. bar 함수 코드 실행
      - `console`은 객체 환경 레코드 `BindingObject`를 통해 전역 객체에서 찾을 수 있음
      - `log`는 `console` 객체가 직접 소유하는 프로퍼티
   8. bar 함수 코드 실행 종료
      - 렉시컬 환경은 실행 컨텍스트에 의해 참조되기는 하지만 **독립적인 객체**다. 객체를 포함한 모든 값은 누군가에 의해 참조되지 않았을 때 비로소 가비지 컬렉터에 의해 메모리 공간의 확보가 해제되어 소멸한다
   9. foo 함수 코드 실행 종료
   10. 전역 함수 코드 실행 종료
